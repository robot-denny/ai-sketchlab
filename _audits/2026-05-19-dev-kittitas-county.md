## Pillar 1 — Modern .NET hygiene

**Score**: 4 / 5

**Rationale**: This is textbook modern ASP.NET Core hosting. `Program.cs` uses `Host.CreateDefaultBuilder` with explicit `ConfigureLogging(.AddOpenTelemetry())`, a clean configuration cascade (`appsettings.json` → `appsettings.{EnvironmentName}.json` → `appsettings.{UserName}.json`), and the critical `UseDefaultServiceProvider(opts => opts.ValidateOnBuild = true)` line that catches missing dependencies at startup rather than at first-request. `Startup.ConfigureServices` registers `IMemoryCache`, distributed cache, and `IHttpClientFactory` cleanly, organizes service registration into `#region Scopped / Singleton / Transiants` blocks, and wires API versioning with both URL-segment and header readers. Logging uses `ILogger<T>` (13 references), there are zero `Console.WriteLine` occurrences, and zero `catch (Exception) { }` swallowed-exception patterns.

**Strengths**:
- `Host.CreateDefaultBuilder` + `ValidateOnBuild = true` in [src/Kittitas.Web/Program.cs](src/Kittitas.Web/Program.cs) — the single best DI hygiene line in either codebase.
- `appsettings.{UserName}.json` per-developer override pattern lets each contributor have local settings without touching shared config.
- `services.AddHttpClient()` registered — sets the team up to add Polly via `AddPolicyHandler` later with one-line additions per client.
- `services.AddApiVersioning(...)` with both URL-segment and header readers — supports clean versioning from day one.
- OpenTelemetry-pumped logging configured at the host level (`AddOpenTelemetry()` in `ConfigureLogging`).
- `using` statements organized into ordered groups (stdlib → Microsoft → framework → project) per the file headers I sampled.

**Weaknesses**:
- **`.editorconfig` exists but is empty** (0 bytes). The file is in the repo, signaling intent, but provides no rules — so editor-side formatting drift will accumulate.
- One detected `.Result` / `.Wait()` sync-over-async pattern. Worth running down and either justifying with a "why" comment or fixing.
- No central package management (no `Directory.Packages.props`). With two projects + multiple Diagram packages, version drift is a real risk.
- The legacy `Startup.cs` split from `Program.cs` is supported but no longer idiomatic for new .NET 6+ apps; the minimal-hosting `WebApplicationBuilder` model would be slightly more conventional today. Not a real problem — just dated relative to brand-new templates.

## Pillar 2 — Architectural separation

**Score**: 4 / 5

**Rationale**: This is the **strongest pillar** for this codebase, and the dimension where it most clearly outperforms most Umbraco sites. The two-project split is the canonical "host vs library" pattern done right: `Kittitas.Web` is the composition root (entry point, frontend, configs); `Kittitas.Features` is a Razor Class Library that holds all business logic + Razor views shipped as embedded resources. Inside `Kittitas.Features`, the organization is feature-folder + technical-folder hybrid: `Blocks/<ElementName>/Views/Index.cshtml` for vertical slicing of block components; `Services/<Domain>/` for service-layer code by domain (`SEO/`, `Rendering/`, `BackofficeUser/`, etc.); a dedicated `Abstractions/` folder for explicit interface seams; `Infrastructure/` for cross-cutting concerns (ContentFinder, ViewModelFactory, TagHelpers); `Controllers/API/<Domain>/` for versioned API endpoints. The dependency direction is enforced by the project boundary itself (RCL can't reference its host).

**Strengths**:
- **Vertical-slice block organization**: each block has its own folder (`AccordionElement/`, `CalloutElement/`, `RichTextElement/`, etc.) containing its own `Views/Index.cshtml`. Adding a block doesn't pollute a shared partials directory.
- **Explicit `Abstractions/` folder** at [src/Kittitas.Features/Abstractions/](src/Kittitas.Features/Abstractions/) with focused interfaces (`IPageSettingService`, `ITemplateCoordinator`, `IWebPage`). Even though some are tiny, the *intent* of exposing seams is correctly signaled.
- **API controllers organized by domain** with per-search-type handlers ([src/Kittitas.Features/Controllers/API/Search/](src/Kittitas.Features/Controllers/API/Search/) has `NewsPressSearchApi`, `GeneralSearchApi`, etc.). Endpoint mapping is done via extension methods (`MapSearchEndpoints`), a clean composition pattern.
- **Services organized by domain** (`BackofficeUser/`, `Rendering/`, `SEO/`, `ThemeService/`) — when adding a new SEO concern, the home is obvious.
- **RCL with embedded resources** for Razor views: in Release builds the views ship from the assembly via `EmbeddedResource` globs + `EmbeddedFileProvider`; in Debug builds a `PhysicalFileProvider` lets you edit-and-reload. Sophisticated production tradeoff.

**Weaknesses**:
- Some `Abstractions/` interfaces are very narrow — `IPageSettingService` has one method (`PageHeading(IPublishedContent currentPage)`). Borderline shallow. Either the interface is set up for future expansion (fine — note the why), or it's just hiding a one-liner (then the deletion test suggests inlining it). Verify on a per-interface basis.
- The `Diagram.*` framework dependencies aren't behind an anti-corruption layer. Pragmatic given how deep Diagram is into the architecture (essentially the Umbraco extension framework here), but worth flagging that a future migration off Diagram (or even a Diagram major-version bump) would touch many files.
- No domain-driven design vocabulary visible — entities are mostly Umbraco published models, not rich domain types. For a content site this is appropriate; for any future modules that handle business workflows (payments, registrations) the codebase isn't set up to keep those models clean.
- Both `Kittitas.Web.sln` (at repo root) and `src/src.sln` exist. Likely an IDE-vs-Cloud-build artifact, but a single source of truth would be cleaner.

## Pillar 3 — Umbraco-version-appropriate patterns

**Score**: 4 / 5

**Rationale**: The Umbraco fundamentals are solid: 4 composers (`LuceneIndexComposer`, `ThemeDropdownComposer`, plus two more), 2 notification handlers, 239 `.uda` schema artifacts committed, schema-as-code discipline visible. The Diagram framework adds additional Umbraco conventions (e.g., `services.AddDiagramUmbracoServices(dbContext)` and `app.UseDiagramUmbraco()`). However, there are two notable gaps relative to current Umbraco 17 best practice: **zero `CachedPartialAsync` usage** (cached partials are the canonical Umbraco performance idiom and the absence is surprising for a 759-commit production codebase), and backoffice extensions still use the older `App_Plugins/` mechanism (`DiagramCustomStyles`, `DiagramSeoDashboard`) rather than the modern Razor Class Library approach.

**Strengths**:
- Composer pattern used consistently for cross-cutting concerns: `LuceneIndexComposer` for search-index tuning, `ThemeDropdownComposer` for backoffice UI customization.
- 239 `.uda` artifacts committed — schema-as-code is taken seriously.
- Diagram framework conventions (`AddDiagramUmbracoServices` / `UseDiagramUmbraco`) add a layer of additional scaffolding beyond stock Umbraco — provides design view engine, view components, SEO redirects middleware, and storage provider abstractions.
- Per-environment config files plus a `appsettings.{UserName}.json` pattern for per-developer overrides — well-suited for an 11-contributor team.
- `Umbraco.Cms.DevelopmentMode.Backoffice` is intentionally present (per CLAUDE.md note: required for InMemoryAuto Models Builder support). The team knows *why* this dependency exists.

**Weaknesses**:
- **Zero `CachedPartialAsync` usage.** This is the largest Umbraco-idiom gap. Navigation, footer, and other expensive partials should be cached; in a 759-commit production codebase the absence reads as either an oversight or an undocumented decision. If deliberate, document the why; if not, fix it (this is one of the highest-ROI single changes available).
- **No `.uda` pre-commit guard.** With 11 contributors and a Cloud-deployed schema, accidentally committing autogenerated `.uda` drift is a real risk. A pre-commit hook (or equivalent CI check) would catch it before it reaches Live.
- **App_Plugins backoffice extensions** (`DiagramCustomStyles`, `DiagramSeoDashboard`) use the older mechanism. The modern RCL-based backoffice extension pattern is more discoverable, ships better, and supports TypeScript more cleanly. Not a P0 for a working extension, but a P2 for any new extension work.
- Notification-handler density is on the low side (2 handlers for a 208-`.cs`-file codebase). Either the project doesn't have many content-lifecycle workflows (plausible for a county site), or some logic that *should* be in notification handlers is sitting elsewhere. Worth a sample inspection.

## Pillar 4 — Headless suitability

**Score**: 4 / 5

**Rationale**: The Vue Islands frontend pattern already proves a significant partial-decoupling. `src/Kittitas.Web/src/main.js` uses `import.meta.glob` to lazy-load both vanilla JS components and Vue SFCs based on `data-*` attributes present in rendered HTML. Vue itself is only imported when a Vue hook is found on the page. Components are mounted from server-rendered HTML, so the site is still Razor-SSR — but the interactive layer is fully decoupled and the architecture is well-positioned for moving more of the rendering to Vue if/when desired. API controllers expose Search and ContentListing under versioning, providing an existing API surface for any future SPA frontend.

**Strengths**:
- **Vue Islands pattern** (PascalCase component → kebab-case data attribute) auto-mounts components from server-rendered HTML. This is the right modern hybrid pattern.
- `wwwroot/dist` is the production output via Vite, with a custom manifest filename (`vite-manifest.json` rather than `.vite/manifest.json`) to survive Azure/Kudu's dot-prefix-stripping deploy behavior — the *why* is documented in CLAUDE.md.
- Search is exposed via API controller, not Razor view — already headless-shaped.
- `ViewModelFactory` infrastructure in `Kittitas.Features/Infrastructure/` separates view models from `IPublishedContent`, an important seam for headless migration.
- Tailwind v4 utility classes mean the styling is portable to a JSX/Vue/Svelte frontend with minimal changes.

**Weaknesses**:
- Site is still Razor-SSR at the top level. A full headless move would still require establishing a public-site frontend project (the Vue Islands are *enhancements*, not the primary renderer).
- Only 3 `DeliveryApi` references — the framework is less aggressively exposing content via Delivery API than the comparison repo. If headless is a roadmap item, this is the area to invest.
- ContentListing API is solid but per-search-type API handlers (`NewsPressSearchApi`, `GeneralSearchApi`) could be unified behind a more general content-query API if the goal were full headless.

### Headless trajectory

- **Current orientation**: hybrid (Razor-SSR + Vue Islands)
- **Migration readiness**: well-positioned to neutral
- **Migration cost estimate** (if pursued today):
  - Content access refactoring: **S** — `ViewModelFactory` already separates view models from `IPublishedContent`; reusing them for a headless API is straightforward.
  - Business logic extraction from templates: **S** — most of it's already in services (`NavigationService`, `SiteSettingsService`, etc.) rather than Razor.
  - Frontend rewrite or stand-up: **M** — the Vite/Vue infrastructure is already in place; you'd be promoting the Islands to a full SPA, not building from scratch.
- **Key changes that would be required**:
  - Expand the Delivery API surface (add page-level endpoints, not just Search/ContentListing).
  - Promote Vue Islands to top-level page renderers (Nuxt or a Vue 3 + Vue Router SPA).
  - Move dictionary item access from `Umbraco.GetDictionaryValue` calls in Razor to a Delivery API surface.
  - Resolve image URLs and asset manifests through the API rather than via Razor helpers.
- **What you'd want to preserve**:
  - The Vite manifest discipline (custom filename, base path) — applies equally to a SPA.
  - The Vue Islands components — they become the building blocks of the SPA.
  - The Diagram view-component infrastructure — usable from a headless renderer with minor adaptation.

## Recommendations

Prioritized for the *growing* lifecycle stage (11 contributors, 759 commits, 8 months old, active velocity). Lock in operational hygiene now before scale makes it painful.

### P0 — Do these soon

#### P0.1 — Stand up a CI pipeline
- **Rationale**: 11 contributors at 759 commits with **no automated gate before merge** is the single highest-risk operational gap. A GitHub Actions workflow that runs `dotnet build` + (eventually) `dotnet test` + `npm run build` on every PR catches regressions before Cloud rejects them at deploy.
- **Scope**: S
- **Lifecycle-stage appropriateness**: growing P0 — should have shipped months ago; should ship this sprint.
- **Pillars improved**: 6, 7

#### P0.2 — Adopt `CachedPartialAsync` on navigation, footer, and other expensive partials
- **Rationale**: Zero usage of the canonical Umbraco caching idiom in a 759-commit production codebase is anomalous. Navigation, footer, breadcrumbs, and any expensive ViewComponent should be cached with explicit TTLs. Single highest-ROI Umbraco-pattern adoption available.
- **Scope**: S to introduce per partial; M to audit all candidates.
- **Lifecycle-stage appropriateness**: growing P0.
- **Pillars improved**: 3, 7

#### P0.3 — Add an xUnit test project
- **Rationale**: Strong architectural separation (Pillar 2 = 4) is wasted without unit tests to exercise the seams. Start with `PageSettingsService`, `NavigationService`, and `BlockListMapperService` (pure-logic services). Establish the pattern; tests will follow.
- **Scope**: S to set up; M to populate meaningfully.
- **Lifecycle-stage appropriateness**: growing P0 — the architecture is begging to be tested.
