## Executive summary

This is a well-curated Umbraco 17 site that punches above its weight on **agentic-collaboration scaffolding** and **schema discipline**, but is thin on **architectural separation** and **operational resilience**. The C# surface is small (11 files outside Views) and uses Umbraco's idioms cleanly — Composers, notification handlers, DI, structured logging — but business logic concentrates in Razor templates rather than services, which both ceilings the architecture-separation score and constrains future headless work. The codebase is in a sweet spot for *foundational* improvements: lifecycle classification is **growing**, so locking in patterns now is still cheap. The headline recommendations are: add a thin service layer behind the search view, add at least one health-check + retry policy for the AI/OpenAI outbound, and add unit-test scaffolding before the next major feature lands.

## Pillar 1 — Modern .NET hygiene

**Score**: 4 / 5

**Rationale**: The bootstrap is clean and idiomatic for Umbraco 17. `Program.cs` is six lines of meaningful wiring (`CreateUmbracoBuilder → AddBackOffice → AddWebsite → AddComposers → Build`) plus controller registration via `AddApplicationPart`. Logging goes through `ILogger<T>` (structured) and Serilog is configured via `appsettings.json`. There are zero `.Result` / `.Wait()` blocking calls and zero `Console.WriteLine` occurrences outside startup. The biggest gap is the absence of explicit nullable / warnings-as-errors enforcement — not detected at the `.csproj` level.

**Strengths**:
- Composition is consolidated in `IComposer` implementations rather than sprawled in `Program.cs`. See [SearchComposer.cs](src/UmbracoProject/SearchComposer.cs), [AssignMembersToPremiumRoleComposer.cs](src/UmbracoProject/AssignMembersToPremiumRoleComposer.cs), [PaletteServiceComposer.cs](src/HelloWorld/PaletteServiceComposer.cs).
- Structured logging with named parameters used correctly: `_logger.LogInformation("Automatically assigning member {MemberId} to role {RoleName}", member.Id, RoleName)` in [AssignMembersToPremiumRoleHandler.cs:30-32](src/UmbracoProject/AssignMembersToPremiumRoleHandler.cs#L30-L32).
- Specific-exception catches rather than `catch (Exception)`: `catch (FormatException ex)` in [PaletteService.cs:54](src/HelloWorld/PaletteService.cs#L54) and `catch (Win32Exception ex)` in [ImageGeneratorController.cs:156](src/HelloWorld/ImageGeneratorController.cs#L156).
- `.editorconfig` present at repo root — mechanical formatting is auto-enforced.

**Weaknesses**:
- Nullable annotations and warnings-as-errors aren't visible at the `.csproj` level (no `<Nullable>enable</Nullable>` / `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` detected). Hard to tell if this is a deliberate choice or an oversight; either way it's a cheap improvement at growing stage.
- No central package management (no `Directory.Packages.props`). With only two projects this is marginal, but if a third project is added (a test project would be the obvious one), centralizing will avoid version drift.

## Pillar 2 — Architectural separation

**Score**: 2 / 5

**Rationale**: Layering is absent. There is no `Domain/`, `Application/`, `Infrastructure/`, or feature-folder organization in either project. The 11 hand-written `.cs` files outside Views split as: 3 Composers, 1 notification handler, 3 helpers, 1 service (`PaletteService`), 1 controller (`ImageGeneratorController`), 1 `Program.cs`, 1 `Constants.cs`. There are no repository or anti-corruption interfaces — `IPublishedContentQuery`, `IContentService`, `IContentTypeService`, `IMemberService` are consumed directly. Business logic that *should* be in a service (the search pipeline, the CLI invocation) lives in Razor or in the controller respectively.

**Strengths**:
- `PaletteService` is a textbook deep module: small public surface (`GetPaletteConfigJson()`), significant internal logic (content lookup, hex parsing, palette assembly, error handling), behind a clean DI seam. See [PaletteService.cs](src/HelloWorld/PaletteService.cs).
- The notification handler pattern is used at the right level: a domain event (`MemberSavedNotification`) is handled by a single-purpose class that depends only on what it needs. See [AssignMembersToPremiumRoleHandler.cs](src/UmbracoProject/AssignMembersToPremiumRoleHandler.cs).
- DTO types (`PaletteConfigDto`, `PaletteEntryDto`) separate JSON-shaped models from `IPublishedContent`.

**Weaknesses**:
- [search.cshtml](src/UmbracoProject/Views/search.cshtml) carries substantial business logic in the view: a two-backend search pipeline (Examine for short queries, AI semantic for long), a 5-minute memory cache, fallback chain, document-type filtering, XSS-safe encoding. The comments are excellent, but the *location* is wrong — this belongs in a `SearchService` injected into a tiny view that just renders results. Moving it would also unblock the Delivery API exposure of the same search behavior.
- [ImageGeneratorController.cs](src/HelloWorld/ImageGeneratorController.cs) spawns an external CLI subprocess (`tsx scripts/image-generator/src/cli.ts`) directly. This couples the backoffice API to a sibling repo path, the host file system, and a Node binary. There is no anti-corruption boundary — a "process spawner" abstraction with a fake/stub would let this be tested and would make swapping for an in-process implementation cheaper.
- No repository or query-object pattern. Razor partials and the controller reach into Umbraco's published-content cache directly.

## Pillar 4 — Headless suitability

**Score**: 3 / 5

**Rationale**: The framework is set up to support headless reads: `AddDeliveryApiSearch()` is composed, the AI Search infrastructure is wired through Core, and the Vite-built backoffice extension demonstrates that a JS build pipeline already exists. However, the *site itself* is rendered server-side via Razor, with significant business logic embedded in templates ([search.cshtml](src/UmbracoProject/Views/search.cshtml), some block components). Migration toward a headless front-end would require extracting the in-Razor business logic to services and standing up a separate frontend project.

**Strengths**:
- Delivery API is enabled at the framework level (14 references across composers and config).
- A frontend build (Vite) already exists for the backoffice extension at [src/HelloWorld/Client/](src/HelloWorld/Client/), proving the team can run a parallel JS-side toolchain.
- Block components are *mostly* presentational — most of `Views/Partials/blocklist/Components/` are clean. The exception is templates that fetch data inline (a few do).
- Content models are typed via the auto-generated `Umbraco.Cms.Web.Common.PublishedModels` — useful for any future Delivery API consumer.

**Weaknesses**:
- The search experience is built around Razor + IMemoryCache. Reproducing it for headless would require a `SearchController` or minimal-API endpoint plus a `SearchService` doing what the view currently does.
- No separate frontend project consumes Umbraco content yet. The Vite project under `HelloWorld/Client/` builds the backoffice extension, not a public-site frontend.
- Dictionary lookups (`Umbraco.GetDictionaryValue("Search.Placeholder")` etc.) are bound to the Razor view — they'd need Delivery API surfacing for a headless equivalent.

### Headless trajectory

- **Current orientation**: hybrid
- **Migration readiness**: neutral
- **Migration cost estimate** (if pursued today):
  - Content access refactoring: **M** — most page templates use typed models cleanly, but search and any other "computed view" templates need extraction.
  - Business logic extraction from templates: **M** — the search pipeline is the main offender. A handful of block components also do mild computation.
  - Frontend rewrite or stand-up: **L** — no public-site frontend exists yet; this is the bulk of the cost.
- **Key changes that would be required**:
  - Extract a `SearchService` from `search.cshtml` and expose it via a Delivery API endpoint (or a minimal-API controller).
  - Audit `Views/Partials/blocklist/Components/` for any block that fetches its own data; move fetches to a parent ViewModel or service.
  - Establish how dictionary items and other CMS-driven UI strings are exposed for the frontend.
  - Decide on the frontend stack and stand it up as a sibling project (e.g. `src/Web/` or a separate repo).
- **What you'd want to preserve**:
  - The block-component organization and naming — that maps cleanly to a JSX/Vue/Svelte component library.
  - The schema-as-code discipline (`.uda`) — it pays off as much for headless as for traditional.
  - The `Umbraco.AI.Search` integration — exposing it through Delivery API is straightforward once the view-side logic is extracted.

**Score**: 3 / 5

**Rationale**: Scalability is helped by deliberate caching (navigation/footer partials use `Html.CachedPartialAsync`, search results are memory-cached for 5 minutes) and the AI Search infrastructure already chunks/embeds at publish time rather than on each query. **Strengths**:

**Weaknesses**:

## Recommendations

Prioritized using the *growing* row of the lifecycle-stage matrix: "lock in good patterns before they ossify."

### P0 — Do these soon

#### P0.1 — Stand up a CI pipeline
- **Rationale**: 4 contributors, no automated gate before push. A simple GitHub Actions workflow that runs `dotnet build` + `dotnet test` + `npx playwright install --with-deps && npx playwright test --project=chromium --grep-invert @flaky` would catch most regressions before they reach Cloud's build.
- **Scope**: S
- **Lifecycle-stage appropriateness**: greenfield/growing P0 — *cheap now, increasingly expensive to add later.*
- **Pillars improved**: 6, 7

#### P0.2 — Extract `SearchService` from `search.cshtml`
- **Rationale**: The search pipeline (Examine-vs-AI routing, 5-minute cache, XSS-safe encoding, document-type filtering) is genuinely interesting business logic, currently trapped in a Razor view. Extracting it to an injectable `SearchService` lets it be unit-tested, exposed via Delivery API for any future headless frontend, and reused if a typeahead/autocomplete is added later. The view becomes a 30-line renderer that calls the service.
- **Scope**: M
- **Lifecycle-stage appropriateness**: growing P0 — *the search code is mature enough to know what the contract should be, but young enough that nothing else has built on its current shape.*
- **Pillars improved**: 2, 4, 7

#### P0.3 — Add an xUnit test project
- **Rationale**: 0 unit tests today means every refactor is risky. The first test project doesn't have to cover everything — start with `PaletteService` (pure unit) and the future `SearchService` (with `IPublishedContentQuery` mocked). Establishes the muscle and the pattern.
- **Scope**: S to set up; M to populate meaningfully.
- **Lifecycle-stage appropriateness**: growing P0 — *every week without tests adds drag.*
- **Pillars improved**: 6, 7

### P1 — Worth tackling next

#### P1.1 — Add Polly to AI / OpenAI outbound calls
- **Rationale**: The AI Search and Anthropic/OpenAI connections are external dependencies with transient failure modes. Polly via `HttpClient`-level `AddPolicyHandler` (exponential backoff + circuit breaker on the OpenAI client) would prevent a 30-second OpenAI hiccup from becoming a 30-second user-visible search delay.
- **Scope**: S
- **Lifecycle-stage appropriateness**: growing P1.
- **Pillars improved**: 6

#### P1.2 — Add a `/health` endpoint
- **Rationale**: One-liner: `app.MapHealthChecks("/health");` plus `AddHealthChecks().AddDbContextCheck<UmbracoDbContext>()` (or equivalent). Lets uptime monitors and Cloud-side health checks distinguish "app booted" from "app responsive."
- **Scope**: S
- **Lifecycle-stage appropriateness**: growing P1.
- **Pillars improved**: 6

#### P1.3 — Replace the CLI-shellout pattern in `ImageGeneratorController` with a `IImageGenerator` interface
- **Rationale**: Today the controller spawns `npx tsx scripts/image-generator/src/cli.ts` as a subprocess. This is a brittle seam — it's untestable, ties the backoffice extension to the host filesystem, and surfaces shell-quoting concerns. An `IImageGenerator` interface with two implementations (`CliImageGenerator` for now, `InProcessImageGenerator` later) gives a clean test seam and a migration path.
- **Scope**: M
- **Lifecycle-stage appropriateness**: growing P1 — *the seam is small enough today that it can be introduced without an upheaval.*
- **Pillars improved**: 2, 6, 7

#### P1.4 — Enable nullable reference types and `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>`
- **Rationale**: Cheap insurance against a class of common bugs. Most modern .NET project templates do this by default; this codebase predates that.
- **Scope**: S (mechanical) — but expect some real fixes to surface.
- **Lifecycle-stage appropriateness**: growing P1.
- **Pillars improved**: 1, 7

### P2 — Nice to have

#### P2.1 — Convert the informal ADRs in CLAUDE.md to a `docs/adr/` folder
- **Rationale**: The *Pinned betas — do not float*, *Schema drift root causes*, and *AI schema deployment bootstrapping* sections are real architectural decisions that deserve to be linkable in PRs and discoverable in their own right. Keep CLAUDE.md as the index; move the content.
- **Scope**: S
- **Lifecycle-stage appropriateness**: growing P2.
- **Pillars improved**: 5

#### P2.2 — Add OpenAPI / Swagger to the `ImageGeneratorController`
- **Rationale**: The backoffice extension exposes a small API surface (`/palettes`, `/articles`, `/categories`, `/generate/batch`, `/generate/{id}`). Documenting it with `AddSwaggerGen` would help any future TypeScript client (the auto-generated OpenAPI client mentioned in CLAUDE.md presumably already does some of this — confirm and extend).
- **Scope**: S
- **Lifecycle-stage appropriateness**: growing P2.
- **Pillars improved**: 5

#### P2.3 — Introduce central package management (`Directory.Packages.props`)
- **Rationale**: With two projects you can get away without it. The first time you add a third project (a test project would be the natural one), centralize first.
- **Scope**: S
- **Lifecycle-stage appropriateness**: growing P2.
- **Pillars improved**: 1

#### P2.4 — Begin a thin OpenTelemetry instrumentation pass
- **Rationale**: Once the AI / Delivery API endpoints see real traffic, request tracing will pay back fast. Start with the OpenTelemetry .NET auto-instrumentation for `HttpClient` and ASP.NET Core; add custom spans around the search pipeline if/when it moves to a service.
- **Scope**: M
- **Lifecycle-stage appropriateness**: growing P2 — *no rush, but worth the seed.*
- **Pillars improved**: 6

## Comparison — umbraco-17-demo-site vs dev-kittitas-county

Both repos are Umbraco 17, both classified as `growing`, but their *shape* could not be more different. The demo site is a small (11 `.cs` files outside Views), tooling-rich, AI-collaboration-forward exploratory codebase. Kittitas is a substantial (208 `.cs` files outside Views), production-grade, professionally-architected client site built on the Diagram framework with 11 contributors at 759 commits over 8 months. Each repo's strengths illuminate the other's weaknesses; together they sketch what a "best of both" Umbraco architecture would look like. **Full Kittitas audit**: [_audits/2026-05-19-dev-kittitas-county.md](_audits/2026-05-19-dev-kittitas-county.md).

### Side-by-side scores

|---|---|---|

| 2 — Architectural separation | **2** | **4** |
| 3 — Umbraco-version-appropriate | **5** | **4** |

### What to learn from each

- **From umbraco-17-demo-site** (what Kittitas should consider adopting):
  - **The `.uda` pre-commit guard pattern**. With 11 contributors and a Cloud-deployed schema, accidental schema drift is a real risk that the pre-commit hook in `.githooks/pre-commit` would catch before push. One file's worth of work.
  - **`CachedPartialAsync` discipline**. The demo site uses it on navigation and footer with explicit TTLs; Kittitas uses it zero times in a 759-commit production codebase. This is the highest-ROI single Umbraco adoption Kittitas could make.
  - **Agentic-collaboration scaffolding**. `.claude/commands/`, `.claude/agents/`, `_specs/_plans/_features/` — for an 11-contributor team, the leverage from agent-native review tooling and spec-driven development would compound quickly.
  - **Umbraco MCP + backoffice-skills plugins**. Wiring these up lets the team's AI assistants understand Umbraco-version-specific best practices and author content programmatically from the CLI.
  - **A real `.editorconfig`**. Kittitas has an *empty* one; the demo site has a working one. Easy fix.
- **From dev-kittitas-county** (what the demo site should consider adopting):
  - **The two-project RCL pattern**. Splitting business logic into a Razor Class Library (`Kittitas.Features`) with embedded-resource Razor views, referenced by a thin host project (`Kittitas.Web`), is the canonical way to keep an Umbraco site's architecture from becoming a flat MVC pile. The demo site's `HelloWorld` is a backoffice extension only; a `<RepoName>.Features` RCL for the public-site logic would be the right next move.
  - **Vertical-slice block organization** (`Blocks/<ElementName>/Views/Index.cshtml`). The demo site's flat `Views/Partials/blocklist/Components/` works for 11 blocks but won't scale; Kittitas' per-folder pattern does.
  - **`ValidateOnBuild = true` and `AddHttpClient()`**. Two single-line additions in `Program.cs`-equivalent that buy real DI hygiene and resilience headroom. The demo site doesn't have them.
  - **OpenTelemetry-pumped logging**. Wiring `AddOpenTelemetry()` into `ConfigureLogging` is a single line that future-proofs observability. The demo site has 0 OTel references.
  - **XML doc comment discipline at scale** (97 files vs 3). For a codebase that wants to be AI-friendly, type-level documentation is leverage the demo site is leaving on the table.
  - **API versioning + domain-organized API controllers**. The Search and ContentListing API controllers with versioning under `Kittitas.Features/Controllers/API/` are exactly the seam that would unblock the demo site's headless trajectory.
  - **An explicit `Abstractions/` folder**. Even when interfaces are narrow, the *intent* of exposing seams is the right signal. The demo site has zero explicit interface seams in C#.

### Pros / cons per pillar

- **Pillar 1 — Modern .NET hygiene (4 vs 4, tie)**. Both codebases use modern idioms cleanly. **Kittitas edges ahead** on `ValidateOnBuild = true` (catches DI bugs at startup), `AddHttpClient()` (sets up Polly cheaply), and OpenTelemetry-pumped logging. **The demo site edges ahead** on a populated `.editorconfig` and zero sync-over-async occurrences (Kittitas has one). Net: a small architectural edge to Kittitas, a small hygiene edge to the demo site. Net winner: **slight Kittitas**.

- **Pillar 2 — Architectural separation (2 vs 4, Kittitas wins decisively)**. This is the largest gap. Kittitas has a two-project RCL split, vertical-slice block folders, an explicit `Abstractions/` folder, domain-organized `Services/`, API controllers separated by domain with versioning, and a `ViewModelFactory` infrastructure. The demo site is functionally a flat MVC layout — 11 C# files outside Views, no layering, no interface seams, business logic in Razor views. The demo site is *fine* for its size; Kittitas demonstrates what a real Umbraco architecture looks like. **Winner: Kittitas.**

- **Pillar 3 — Umbraco-version-appropriate patterns (5 vs 4, demo site wins)**. Both have strong schema discipline (220 vs 239 `.uda` artifacts) and Composer adoption. The demo site wins on three counts: the `.githooks/pre-commit` `.uda` drift guard, the modern RCL-based backoffice extension pattern (vs Kittitas' older App_Plugins), and — most significantly — *any* `CachedPartialAsync` usage. Kittitas has *zero* cached partials in a 759-commit production codebase, which is anomalous. **Winner: demo site.**

- **Pillar 4 — Headless suitability (3 vs 4, Kittitas wins)**. Both are hybrid; Kittitas is more headless-shaped. Its `ViewModelFactory` separates view models from `IPublishedContent`, its Search and ContentListing logic is already in API controllers (not Razor), and its Vue Islands frontend proves a partial-decoupling exists. The demo site's search logic lives in [search.cshtml](src/UmbracoProject/Views/search.cshtml) and would need extraction before any headless work. **Winner: Kittitas.**

- **Pillar 5 — Documentation & onboarding (5 vs 3, demo site wins decisively)**. The demo site is the clearer leader here. It has CLAUDE.md plus `_specs/_plans/_features/`, `.claude/{commands,agents,skills}/`, `docs/`, a working `/check-uda` skill, and a /code-review pipeline. Kittitas has a dense, load-bearing CLAUDE.md and 97 files with XML doc comments — *both impressive* — but the `Readme.md` is the unmodified Umbraco Cloud default referencing a `UmbracoProject` path that doesn't exist, there are no ADRs, no spec/plan/feature scaffolding, and no agentic-collaboration tooling. **Winner: demo site.** (But Kittitas' XML doc comment discipline is a lesson the demo site should adopt.)

- **Pillar 6 — Resilience & operations (2 vs 2, tie)**. Both repos are weak here in different ways. **Kittitas has OpenTelemetry-pumped logging** and `AddHttpClient()` (the seeds for resilience); **the demo site has the `.githooks/` pre-commit infrastructure** and a working `.editorconfig`. Neither has CI, neither has Polly, neither has health checks, neither has unit tests. Net: tied, but they'd be a clear pair of strengths if combined. **Combined winner: a future "best of both."**

- **Pillar 7 — Scalability & refactorability (3 vs 3, tie)**. The architectural-separation gap (Pillar 2) gives Kittitas *more refactor leverage* in principle, but zero unit tests means that leverage is unrealized. The demo site has cached partials but a thin C# surface that doesn't need much refactoring anyway. Both share the unit-test gap. **Net: structurally Kittitas leads, but neither realizes the potential.**

### Headline takeaway

**These two repos are complementary, not competitive.** Kittitas demonstrates what to build *toward* architecturally; the demo site demonstrates what to build *with* in terms of agentic-collaboration tooling, schema discipline, and Umbraco-pattern adoption. The ideal third codebase combines Kittitas' two-project RCL + vertical slicing + Vue Islands + API-versioned controllers with the demo site's `.uda` pre-commit guard + `CachedPartialAsync` discipline + `.claude/commands/agents/skills/` agentic scaffolding + MCP/backoffice-skills wiring + a populated `.editorconfig`. Each codebase would close 80% of its weakest pillar by adopting two-to-three patterns from the other.
