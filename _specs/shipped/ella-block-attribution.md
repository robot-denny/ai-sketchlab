# Spec for Ella Block Attribution

## Summary

The blog is becoming a collaboration surface between the human site author and an AI persona named **Ella**. When the human and Ella co-write an article, each prose block can carry a different voice — and readers need to see, at a glance, whose words they're reading inside the body copy. The article masthead's existing byline is enough when one person wrote the whole thing; it isn't enough when authorship alternates block-by-block within a single essay.

The design system already ships the visual treatment for this. The `.ella` rule set in [article-page.css:49-53](src/UmbracoProject/wwwroot/assets/css/article-page.css#L49-L53) — warm secondary surface, subtle left border in tertiary gray, mono eyebrow with a small bullet-dot, light body weight — was lifted directly from `_design-system-handoff/mocks/D-article.html` during Package B. **Nothing in the site currently triggers it on real block content.** Per-block `Author` properties already exist on `richTextRow`, `imageRow`, and `latestArticlesRow` and render a small trailing `<p class="block-author">— Name</p>` byline today, but there is no concept of "this author is the AI persona" and therefore no way for a partial to know it should switch from a quiet byline tag to the strong inline-note treatment.

This feature wires the existing per-block `Author` picker into the existing `.ella` styling via a single new property on the **Author** document type — `isAi: boolean` — and a thin attribute-driven wrapper in the blocklist orchestrator partial. When a block's author is an `isAi` author *and* the article as a whole is not also by that AI author, the block renders inside the `.ella` callout treatment with the eyebrow "Written by Ella · inline note". When the whole article is by the AI author, the block treatment is suppressed (the masthead byline already does the job) so the "inline note" label never appears in a context where it would be nonsensical. Existing trailing `<p class="block-author">` bylines continue to handle human↔human attribution differences, which the editor cares less about marking strongly.

Choosing the Author doc type as the source of truth (instead of a per-block toggle on every block's Settings element type) keeps the existing Author picker as the single editing surface, makes the flag reusable for future AI personas without a rename (`isAi`, not `isElla`), and avoids a multi-block schema migration.

## Functional Requirements

- The **Author** document type ([document-type__eb9fd2d152584bf8beaba2415b2e365a.uda](src/UmbracoProject/umbraco/Deploy/Revision/document-type__eb9fd2d152584bf8beaba2415b2e365a.uda)) gains a boolean property `isAi` (default `false`). Setting it to `true` on an Author node marks that author as an AI persona.
- The existing per-block `Author` picker on `richTextRow`, `imageRow`, and `latestArticlesRow` is the single editing surface for block attribution. No new per-block setting or composition is added.
- When a block inside an article body has an Author whose `isAi == true` **and** the article's primary author(s) do not all have `isAi == true`, the block renders inside the existing `.ella` visual treatment (warm-stone background, mono eyebrow, subtle left border, light body weight).
- The eyebrow text reads `Written by Ella · inline note` (or the AI author's name in place of "Ella" if the persona is ever renamed). The eyebrow is generated by CSS (`::before`) on the wrapper, not emitted per-partial.
- When the block-level wrapper treatment is active, the existing trailing `<p class="block-author">— Name</p>` byline is suppressed for that block so the same name doesn't appear twice in the same card.
- When the article's primary author is itself an `isAi` author, individual blocks are **not** wrapped in the `.ella` treatment — the masthead byline carries the attribution and the "inline note" label would be misleading.
- The wrapper applies uniformly across all content blocks in the block list (`richTextRow`, `imageRow`, `imageCarouselRow`, `codeSnippetRow`, `alertBanner`, `videoRow`, `latestArticlesRow`, `iconLinkRow`, `colorPaletteBlock`, `typographyShowcaseBlock`, `generalElementsBlock`). Blocks without an `Author` property simply never trigger the wrapper.
- Implementation lives in **one** place — the blocklist orchestrator [Views/Partials/blocklist/default.cshtml](src/UmbracoProject/Views/Partials/blocklist/default.cshtml) — not in each individual block component partial.
- The wrapper is layout-neutral (`display: contents`) so it does not nest a new visible box around blocks whose CSS already targets `.art-body > .richtext`, `.art-body > .image`, etc.
- Article-level concerns out of scope this round: no masthead byline changes ("By You + Ella" co-author formatting), no blog filter chip ("From Ella"), no Author profile-page changes, no search/list card byline updates.

## Possible Edge Cases

- An article has the AI author as one of several primary authors (co-authored). Should the per-block treatment activate or suppress? **Decision needed:** suppress only when *all* article primary authors are AI, activate otherwise.
- A block's Author is set to an AI author, but the block is itself an `alertBanner` or `videoRow` — block types that don't read as prose. Wrapper still applies uniformly per the requirements, but the eyebrow text "inline note" may not fit. Consider a per-block-type eyebrow variant (or simply rely on the eyebrow phrasing being generic enough).
- A block has no Author set (the default). Should it inherit the article's primary author for attribution purposes? **Decision:** yes, by inheritance — no wrapper triggered because the block author equals the article author.
- An editor flips `isAi` on a long-tenured human Author by accident, causing every past article block by that author to suddenly render as inline notes. The flag should be discoverable but not adjacent to common edit paths — likely behind a "Persona" tab on the Author doc type.
- An AI author is deleted from the site while published articles still reference it. Blocks fall back to the trailing byline tag (or no attribution) without errors.
- The eyebrow text "Written by Ella · inline note" exceeds the column width at narrow viewports. Should we shorten to "Ella · inline note" at small breakpoints? (See open questions.)
- A future second AI persona is added. The system should treat any `isAi == true` author the same, with the persona's display name swapped into the eyebrow text dynamically.
- A block is hidden via the existing `HideProperty` setting; the wrapper should not emit for hidden blocks.
- The trailing `<p class="block-author">` is *also* suppressed when the wrapper is active, but the underlying property still renders if other CSS expects it. Need to confirm no other partials depend on `.block-author` visibility.
- An article entirely by Ella has an image block whose Author was explicitly set to the human (rare reverse case). Wrapper would activate with eyebrow "Written by [Human] · inline note" — which is the symmetric correct answer but currently out of scope for treatment design. **Decision:** activate the wrapper but use a neutral eyebrow without "inline note" (or defer entirely until requested).
- The AI Copilot generates content into a block without setting Author. The block ships with no AI attribution until manually corrected. Possible follow-up: teach the Copilot to stamp the AI Author on generated blocks.

## Acceptance Criteria

1. Author doc type has an `isAi` boolean property; an Author node can be flagged as an AI persona via the backoffice without code changes.
2. A block whose Author is `isAi` and whose article's primary author(s) are not all `isAi` renders inside the `.ella` callout treatment.
3. The "Written by [Name] · inline note" eyebrow text appears exactly once per such block, at the top of the wrapper, generated by CSS rather than partial markup.
4. The existing trailing `.block-author` byline is hidden when the wrapper is active.
5. When the article's primary authors are all `isAi`, blocks render with no wrapper, no eyebrow, and no per-block AI styling — the masthead byline carries the attribution.
6. The implementation touches one orchestrator partial, one CSS file, and the Author doc type — not the eleven individual block component partials.
7. Existing per-block `Author` picker behavior on `richTextRow`, `imageRow`, and `latestArticlesRow` is preserved for human↔human attribution differences (the trailing byline tag still renders in that case).
8. `/check-uda` reports no conflicts for the Author doc-type change before push.
9. A Playwright E2E spec asserts the four-row truth table below.

| Article author | Block author | Wrapper attribute | Eyebrow text | Trailing byline |
|---|---|---|---|---|
| Human | Human | (none) | (none) | renders if author differs from article |
| Human | AI (Ella) | `data-attributed-to="ella"` | `Written by Ella · inline note` | hidden |
| AI (Ella) | AI (Ella) | (none) | (none) | hidden (article-wide) |
| AI (Ella) | Human (rare) | (decision deferred — see edge cases) | n/a | n/a |

## Scenarios (Draft)

### Rule: AI author flag drives per-block treatment, not the editor's per-block toggle

```scenario
Scenario: Marking an Author as an AI persona via isAi flag
  Given an Author content node named "Ella"
  And the editor enables the "isAi" property on the Author
  When the Author is saved and published
  Then any block in any article that picks "Ella" as its Author is eligible for the AI treatment
  And no per-block setting needs to be flipped to enable that treatment
```

### Rule: Block-level AI treatment activates only when the article isn't entirely by the AI author

```scenario
Scenario: Human-led article with an Ella-attributed prose block renders the inline-note treatment
  Given an article whose primary author is a human
  And a rich-text block in that article whose Author is "Ella" (isAi = true)
  When a reader views the article
  Then the rich-text block renders inside the .ella callout treatment
  And the eyebrow reads "Written by Ella · inline note"
  And the trailing "— Ella" block-author byline does not appear

Scenario: All-Ella article renders blocks without per-block treatment
  Given an article whose primary author is "Ella" (isAi = true)
  And every block in that article either has no Author set or picks "Ella"
  When a reader views the article
  Then no block renders the .ella wrapper treatment
  And no block renders an "inline note" eyebrow
  And the article masthead byline reads "By Ella"
```

### Rule: Wrapper application is uniform across block types and layout-neutral

```scenario
Scenario: Image block attributed to Ella inside a human-led article
  Given an article whose primary author is a human
  And an image block in that article whose Author is "Ella"
  When a reader views the article
  Then the image block renders inside the .ella callout treatment
  And surrounding blocks not attributed to Ella render unchanged

Scenario: Existing per-component layout is not disturbed by the new wrapper
  Given an existing article without any AI-attributed blocks
  When a reader views the article
  Then no .ella wrapper is emitted
  And the rendered HTML structure of each block is unchanged from before the feature shipped
```

### Rule: Human↔human attribution differences keep the existing quiet byline tag

```scenario
Scenario: A block by a guest human contributor in an otherwise human-led article
  Given an article whose primary author is "Author A" (isAi = false)
  And a rich-text block in that article whose Author is "Author B" (isAi = false, different from Author A)
  When a reader views the article
  Then the block renders with no .ella wrapper
  And the existing trailing "— Author B" block-author byline appears as today
```

## Open Questions

1. **Eyebrow text at narrow widths.** Do we keep `Written by Ella · inline note` everywhere, or collapse to `Ella · inline note` below a breakpoint? Mock uses the long form.--collapsing is ideal
2. **Wrapper element vs. attribute on existing wrapper.** The plan recommends a *new* wrapper div with `display: contents` to keep the implementation in one partial. The alternative — adding `data-attributed-to` to the existing per-component wrapper (`.richtext`, `.image`, etc.) — avoids a new DOM node but contradicts the "one place" goal and requires touching every block partial. Confirm the new-wrapper approach is acceptable. --confirmed
3. **Co-authored article semantics.** When an article has multiple primary authors and some are `isAi`, do per-block AI wrappers activate (default proposed: yes, suppress only when *all* article authors are AI), or should the rule be stricter? --proposed rule is accepted
4. **Reverse case (AI article, human-attributed block).** Out of scope for visual design this round, but the symmetric scenario will eventually need treatment. Capture a placeholder rule now, or defer entirely? -defer for now
5. **AI Copilot stamping.** Should the Umbraco backoffice AI Copilot automatically set the Author to a configured AI Author node when it generates a block? Follow-up feature or part of this one? --follow up feature a copilot capabilities will change soon. we can revisit then.
6. **Where does `isAi` live — Author doc type directly, or a small composition?** Adding directly is one less artifact; a composition (`AIPersonaProperties`?) follows the established `HideProperty` / `SpacingProperties` precedent in [document-type__da15dc4343f645f6bda81fd17a49d25c.uda](src/UmbracoProject/umbraco/Deploy/Revision/document-type__da15dc4343f645f6bda81fd17a49d25c.uda). --follow established composition model UNLESS the plan provides strong benefits not to
7. **Eyebrow persona name source.** "Ella" is hardcoded in the design treatment text. The implementation should pull the AI author's display Name from the Author content node so a future second persona renders correctly without code changes.--confirmed

## Testing Guidelines

Create a Playwright E2E spec at `tests/e2e/blocks/ellaBlockAttribution.spec.ts` covering:

- An Author node with `isAi = true` is recognizable from its content (use Management API lookup).
- A human-authored article with an Ella-attributed `richTextRow` renders the wrapper attribute, the eyebrow text, and suppresses the trailing byline.
- An Ella-authored article with Ella-attributed blocks renders no wrapper, no eyebrow.
- A human-vs-human attribution difference renders the existing `.block-author` byline tag and no wrapper.
- The wrapper applies to a non-rich-text block (e.g. `imageRow`) in a human-led article when attributed to Ella.
- Regression: an article with no AI-attributed blocks renders byte-identical HTML around each block compared to before the feature.

Follow the E2E Test Resilience Rules in `CLAUDE.md` (dynamic UUID lookup, clean stale test data, regex-tolerant CSS assertions, prefer browser assertions over file-content assertions).

Unit-level coverage: if a `Razor` extension method is introduced to resolve `(blockAuthor, articleAuthors) → wrapperState`, add a small xUnit test exercising the four-row truth table directly.
